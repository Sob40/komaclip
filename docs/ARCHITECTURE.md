# KomaClip Architecture

KomaClip is a Rails SaaS for creating social promo clips from comics and webtoons.

## Core Decisions

- Rails owns users, sessions, projects, credits, permissions, and admin.
- Pixi is the single visual source of truth for preview and final rendering.
- Sidekiq handles render, AI, billing webhook, storage, and mail jobs.
- Polar is the Merchant of Record for checkout, tax, invoices, and subscriptions.
- PostgreSQL is the product database.
- Cloudflare R2 or S3 stores original assets and generated outputs.
- Render is the initial deployment target.

## Production Shape

```text
Browser
  -> Rails web
    -> PostgreSQL
    -> Redis / Valkey
    -> Sidekiq workers
    -> R2 / S3 private storage
    -> Polar webhooks and customer portal
    -> AI providers
```

## Security Rules

- Never trust client-side entitlement, render, billing, or quota state.
- Grant paid access only from verified Polar webhooks.
- Store originals and renders privately; serve them with short-lived signed URLs.
- Keep provider keys in server environment variables or credentials only.
- Make webhook and render job handlers idempotent.
- Require ownership checks on every project, asset, clip, and render output.
- Rate-limit login, registration, checkout, upload, AI, and render endpoints.

## Render Pipeline

```text
Project assets
  -> montage plan
  -> Pixi scene contract
  -> Sidekiq render job
  -> headless Pixi frame capture
  -> FFmpeg MP4
  -> private object storage output
```

The browser can produce fast previews, but paid exports are generated by workers.
